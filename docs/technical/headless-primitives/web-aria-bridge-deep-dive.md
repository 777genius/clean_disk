# Web ARIA Bridge Deep Dive

## Status

Future adapter constraints. Measure before implementing.

- Flutter web accessibility:
  https://docs.flutter.dev/ui/accessibility/web-accessibility
- Flutter assistive technologies:
  https://docs.flutter.dev/ui/accessibility/assistive-technologies

  https://www.w3.org/WAI/ARIA/apg/patterns/

  https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Roles

  https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Guides/Live_regions

## Core Decision

Core Headless exposes semantic intents, not raw ARIA strings. Web ARIA bridge is
an optional adapter if Flutter web semantics are insufficient.

## Flutter First

Default implementation:

- Flutter `Semantics`;
- `SemanticsService.announce` through effects when required;
- focus through Flutter Focus system;
- web accessibility tree generated by Flutter.

Bridge only exists after measurement proves a gap.

## Measurement Matrix

Screen readers:

- VoiceOver + Safari;
- VoiceOver + Chrome;
- NVDA + Firefox;
- NVDA + Chrome;
- keyboard-only without screen reader.

Scenarios:

- TreeGrid row focus;
- TreeGrid cell focus;
- virtualized row indexes;
- sorted header;
- context menu open/restore;
- modal dialog trap;
- tooltip description;
- status polite announcement;
- alert/assertive announcement;
- splitter semantic value.

## Bridge Strategies

Option 1 - Flutter Semantics only:

- lowest risk;
- no DOM duplication;
- may be insufficient for complex `treegrid`.

Option 2 - invisible ARIA mirror:

- maps semantic intents to hidden accessible DOM;
- high risk of duplicate announcements;
- must never expose raw sensitive paths unexpectedly.

Option 3 - targeted role patches:

- add minimal web-only ARIA for specific missing relationships;
- best if technically feasible;
- must be isolated behind adapter.

Recommendation:

```text
measure -> choose targeted patch if needed -> avoid full mirror unless forced
```

## Privacy Rules

Semantic labels can contain file/folder names. Therefore:

- do not log raw semantic labels;
- do not put raw paths in diagnostic bundles without redaction;
- do not expose daemon tokens or session ids;
- avoid stable DOM ids derived from paths.

## Compatibility Rules

- bridge is web-only;
- bridge is optional;
- bridge can be disabled by capability;
- unknown role falls back to Flutter Semantics;
- app can detect degraded accessibility mode;
- tests document browser/screen-reader versions.

## Stop Rules

- Do not promise full ARIA parity before measurement.
- Do not expose browser DOM ids as Headless identity.
- Do not duplicate interactive DOM that conflicts with Flutter.
- Do not make desktop/mobile depend on web-only bridge behavior.
